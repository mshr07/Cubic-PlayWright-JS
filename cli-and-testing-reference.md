# CLI And Testing Reference

This document gives freshers a single place to find:

- JavaScript CLI commands used in this repository
- Playwright CLI commands used in this repository
- Core testing lifecycle definitions
- Common test type definitions
- Basic test case design technique definitions

## JavaScript CLI Commands

These commands help learners run JavaScript examples from the repository.

### Install Project Dependencies

```bash
npm install
```

Purpose:
Installs all project dependencies from `package.json`.

### Run A Single JavaScript File

```bash
node javascript-foundation/01-basics/variables.js
```

Purpose:
Runs one JavaScript topic file directly using Node.js.

### Run The Practical JavaScript Demos

```bash
npm run foundation:run
```

Purpose:
Runs the practical tester-focused JavaScript examples in the foundation folder.

### Run A Specific Async JavaScript Example

```bash
node javascript-foundation/09-asynchronous-javascript/async-await.js
```

Purpose:
Runs a focused async/await example.

### Run ESLint

```bash
npm run lint
```

Purpose:
Checks JavaScript quality and common coding issues.

## Playwright CLI Commands

These commands help learners run browser and API automation suites.

### Install Playwright Browsers

```bash
npx playwright install
```

Purpose:
Downloads the browser binaries required by Playwright.

### Install Specific Browsers

```bash
npx playwright install chromium firefox webkit
```

Purpose:
Installs the three supported Playwright browser engines explicitly.

### Run The Main Playwright Suite

```bash
npm test
```

Purpose:
Runs the Playwright topic suites using the repository config.

### Run In Headed Mode

```bash
npm run test:headed
```

Purpose:
Runs browser tests with a visible browser window.

### Run In Debug Mode

```bash
npm run test:debug
```

Purpose:
Launches Playwright in debug mode for stepping through tests.

### Run Only Locator Tests

```bash
npm run test:locators
```

Purpose:
Runs only the locator-related Playwright test folder.

### Run API Tests Inside The UI/Network Track

```bash
npm run test:api
```

Purpose:
Runs the API/network examples inside `playwright-topic-suites`.

### Run The Dedicated API Testing Folder

```bash
npm run test:api-suites
```

Purpose:
Runs the API-only learning suite from `api-testing-suites`.

### Run Smoke Tests

```bash
npm run test:smoke
```

Purpose:
Runs tests tagged as smoke checks.

### Open The HTML Report

```bash
npm run report
```

Purpose:
Opens the HTML report generated by the Playwright topic suite run.

### Run One Specific Spec File

```bash
npx playwright test -c playwright-topic-suites/playwright.config.js playwright-topic-suites/tests/03-locators-and-selectors/get-by-role.spec.js
```

Purpose:
Runs one specific Playwright spec file.

### Run One Specific Browser Project

```bash
npx playwright test -c playwright-topic-suites/playwright.config.js --project=chromium
```

Purpose:
Runs tests only on one browser project.

### Run The Dedicated API Config Directly

```bash
npx playwright test -c api-testing-suites/playwright.api.config.js
```

Purpose:
Runs the standalone API automation folder directly.

## SDLC And STLC

### SDLC

SDLC stands for Software Development Life Cycle.

Definition:
It is the complete process followed to plan, design, build, test, deploy, and maintain software.

Common stages:

- Requirement gathering
- Design
- Development
- Testing
- Deployment
- Maintenance

Why it matters:
Testers work inside the SDLC, not outside it. Good testers understand where quality fits from requirements to release.

### STLC

STLC stands for Software Testing Life Cycle.

Definition:
It is the testing-specific process followed to plan, prepare, execute, and close testing activities.

Common stages:

- Requirement analysis
- Test planning
- Test case design
- Test environment setup
- Test execution
- Defect reporting and retesting
- Test closure

Why it matters:
It gives structure to testing work and helps teams know what to do at each testing stage.

## Test Types

### Unit Testing

Definition:
Testing the smallest testable part of the application, such as a function, method, or class, in isolation.

Goal:
To verify that one small piece of logic behaves correctly.

Example:
Testing whether a function correctly calculates total price.

Why it matters:
Unit tests are fast and help catch logic defects early.

### Integration Testing

Definition:
Testing how two or more modules, services, or components work together.

Goal:
To verify that combined parts interact correctly.

Example:
Testing whether the UI sends the correct request to the API, or whether the API correctly writes to the database.

Why it matters:
Many bugs happen not inside one module, but at the connection point between modules.

### System Testing

Definition:
Testing the complete system as a whole in an environment close to real usage.

Goal:
To verify that the full product works end to end.

Example:
Testing login, product search, cart, payment, and order confirmation as one full system flow.

Why it matters:
It validates that the integrated product meets the business requirement.

### Regression Testing

Definition:
Re-testing existing functionality after changes to ensure nothing previously working has broken.

Goal:
To detect side effects caused by new development, bug fixes, or configuration changes.

Example:
After adding a new login feature, running old dashboard and profile tests again.

Why it matters:
Regression testing protects product stability over time.

### Black-Box Testing

Definition:
Testing the system based only on inputs and outputs, without knowing the internal code structure.

Goal:
To validate behavior from the user or business point of view.

Example:
Entering valid and invalid login credentials and checking the result.

Why it matters:
This is how most functional, UI, and many API tests are designed.

### White-Box Testing

Definition:
Testing the system with knowledge of the internal code, logic, paths, or implementation.

Goal:
To validate internal logic, branches, conditions, and code-level behavior.

Example:
Testing whether all branches in a validation function are covered.

Why it matters:
It helps improve code coverage and internal quality.

## Test Case Design Techniques

### Boundary Value Analysis

Definition:
A technique where test cases are designed around the edge values of an input range.

Idea:
Bugs often happen at the boundaries, not just in the middle.

Example:
If age must be between 18 and 60, test:

- 17
- 18
- 19
- 59
- 60
- 61

Why it matters:
Boundary testing is one of the most effective ways to catch input validation defects.

### Equivalence Partitioning

Definition:
A technique where input data is divided into groups that are expected to behave the same way.

Idea:
Instead of testing every possible value, test one representative from each valid or invalid group.

Example:
If marks allowed are 0 to 100, partitions could be:

- less than 0
- 0 to 100
- more than 100

Why it matters:
It reduces the number of test cases while still giving strong coverage.

### Decision Tables

Definition:
A technique used when output depends on multiple conditions and combinations of conditions.

Idea:
List conditions and resulting actions in a table so all important combinations are considered.

Example:
For an e-commerce discount:

- Premium user + coupon applied
- Premium user + no coupon
- Regular user + coupon
- Regular user + no coupon

Why it matters:
Decision tables are very useful when business rules are based on several input combinations.
